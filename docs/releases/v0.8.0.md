# CK / CalcKernel v0.8.0 Release Notes

This release note is the release-facing summary for the v0.8.0 line. It covers
the rename to CK / CalcKernel and the Phase 22/23 WASM interop hardening work.
It is intentionally conservative: benchmark results are local measurements, not
cross-machine guarantees.

## Highlights

- Package name: `calckernel`.
- CLI command: `ckc`.
- Source extension: `.ck`.
- C ABI prefix: `CK_`.
- `CKWasmArena` is exported from the package root as a JavaScript/TypeScript
  WASM interop helper.
- `createCKWasmArena(instanceOrExports, options?)` can resolve exported memory
  and heap-base metadata for CK / CalcKernel-generated WASM modules.
- CK / CalcKernel WASM now exports additive `__ck_heap_base` metadata. This does
  not change existing function, memory, pointer, or scalar `f64` ABI behavior.
- Official runnable WASM interop examples are available under `examples/wasm`:
  - `examples/wasm/f64-sum`
  - `examples/wasm/f64-axpy`
  - `examples/wasm/pricing-soa`
- Package smoke now covers the `ckc` CLI, JS import, TypeScript import, and a
  minimal WASM interop consumer path.

## WASM Interop Positioning

`CKWasmArena` is not a CK runtime. It is not a generated-code allocator, GC, IO,
strings, or runtime-library feature. It is a host-side helper for JavaScript and
TypeScript code that needs aligned offsets, typed-array views, bulk copy, and
view refresh after `memory.grow`.

Recommended interop shapes:

- use `Float64Array`, `Int32Array`, `Uint32Array`, `BigInt64Array`, or
  `BigUint64Array` views over WASM memory for homogeneous buffers
- use resident memory when data can stay in WASM memory across repeated calls
- use SoA layout for mixed-width batch data when the host can shape input that
  way
- keep output as a WASM memory view when caller ownership allows it
- use `copyOutF64` only when a JS-owned output copy is explicitly required
- keep `DataView` for fallback/debug ABI paths and small byte-exact checks, not
  for high-throughput per-element or per-field hot paths

For money, tax, discounts, POS totals, and pricing-rule calculations, continue
to use `i64` fixed-point data. Do not use `f64` for financial exact amounts.

## Performance Claims

The correct v0.8.0 performance claim is:

- CK WASM compute paths are competitive on the measured pricing and f64
  workloads.
- Batched resident-memory interop can reduce JavaScript-WASM boundary and memory
  marshaling costs.
- `f64-sum` is a recommended read-only/scalar-return example.
- pricing SoA is a recommended fixed-point resident-memory example.
- `f64-axpy` can be close to, and on the current local Phase 22 run slightly
  faster than, the JavaScript `Float64Array` baseline when output stays as a
  WASM memory view.
- DataView fallback/debug paths can be much slower than JavaScript typed-array
  baselines.

Avoid these release claims:

- do not generalize local CK WASM wins into a blanket JavaScript comparison
- do not generalize the measured cases to all JavaScript typed-array workloads
- do not imply that emitting WASM automatically improves workload speed
- do not describe `DataView` as the high-throughput interop path

Benchmark results depend on machine hardware, Node.js/V8, OS scheduling, power
state, system load, hyperfine, and workload size. Benchmark thresholds are not
part of ordinary `pnpm test`.

## Compatibility Notes

- The package is ESM-first. Import the helper with:

```ts
import { CKWasmArena, createCKWasmArena } from "calckernel";
```

- `CKWasmArena` is public for the v0.8.x release-hardening window, but remains
  experimental before v1.0.
- Existing mixed-width AoS + `DataView` examples remain useful as ABI
  walkthroughs and fallback/debug references.
- For high-throughput WASM interop, prefer the official examples under
  `examples/wasm`.

## Out Of Scope

This release does not add f32, SIMD, fast-math, implicit int/float conversion,
`i64/u64` to `f64` casts, f64-to-int casts, f64 math intrinsics, runtime
allocation, GC, IO, strings, checked WASM arithmetic, or checked LLVM
arithmetic.

## Release Validation

Before tagging or publishing, run:

```sh
pnpm typecheck
pnpm build
pnpm test
npm pack --dry-run
```

Also run a real package fresh-install smoke with `npm pack`, verify `ckc`, JS
import, TypeScript import, and WASM interop consumer behavior, then remove the
generated tarball and temporary directory. Release tags, commits, and npm
publishing remain manual steps.
